# klit-flow

Local, offline code-intelligence tool that extracts **dependency graphs** and **screen-navigation (user-flow) graphs** from a mobile app's source code and emits both human-readable and AI-indexable artifacts.

Everything runs on your machine. No source code, symbol names, or file paths ever leave the device.

---

## Features

- **Dependency layer** — import edges and call edges between files/symbols.
- **Flow layer** — screen nodes and screen-to-screen navigation edges extracted from code and declarative nav files (Android XML nav graphs, etc.).
- **Human-readable output** — Markdown docs with YAML frontmatter + Mermaid diagrams.
- **AI-queryable output** — embedded graph DB (LadybugDB) + local hybrid search (BM25 + semantic embeddings via `sentence-transformers`).
- **MCP server** — exposes graph query tools to AI agents over stdio.
- **Optional NL summaries** — 1–2 sentence descriptions generated by a local Ollama model (`--summaries` flag).

---

## Requirements

- Python 3.11+
- (Optional) [Ollama](https://ollama.com/download) for `--summaries` mode

## Installation

### From source (development / macOS)

```bash
# 1. Clone or copy the source
git clone <your-repo-url> klit-flow
cd klit-flow

# 2. Ensure Python 3.11+ is available
python3 --version        # must be 3.11+
# If not: brew install python@3.11

# 3. Create a virtual environment (recommended)
python3 -m venv .venv
source .venv/bin/activate

# 4. Install in editable mode with dev dependencies
pip install -e ".[dev]"

# 5. Install PyTorch (required by sentence-transformers)
pip install torch

# 6. Verify
klit-flow --help

# 7. Confirm everything is green
ruff format --check . && ruff check . && pytest -q
```

On the first `klit-flow analyze` run the embedding model downloads once (~130 MB) and is cached at `~/.cache/huggingface/`. All subsequent runs are fully offline.

### From a release wheel (macOS / Linux)

Transfer `klit_flow-1.0.0-py3-none-any.whl` to the target machine, then:

```bash
pip3 install klit_flow-1.0.0-py3-none-any.whl
```

**Prerequisites**

- Python 3.11+ — on macOS: `brew install python@3.11`
- PyTorch (required by `sentence-transformers`): `pip3 install torch`

The first time the embedding model is used it is downloaded from HuggingFace and cached locally (`~/.cache/huggingface/`). All subsequent runs are fully offline.

Verify the install:

```bash
klit-flow --help
```

> **Tip (macOS):** if `klit-flow` is not found after install, add pip's bin directory to your PATH:
> ```bash
> export PATH="$HOME/Library/Python/3.11/bin:$PATH"
> ```
> Add that line to `~/.zshrc` to make it permanent.

## Quick start

### 1. Index a repository

```bash
klit-flow analyze /path/to/my-android-app --platform android
```

Supported platforms: `android`, `ios`, `react_native`, `flutter`.

The index is written to `.klit-flow/` inside the target repo (gitignored by default).

### 2. Search from the terminal

```bash
klit-flow query "authentication screen"
```

### 3. Expose to AI agents via MCP

```bash
klit-flow serve          # starts the MCP server over stdio
```

MCP tools available: `query`, `context`, `impact`, `flows`, `cypher`.

Add klit-flow to your MCP client (e.g. Claude Desktop `claude_desktop_config.json`), running it from inside the target repo:

```json
{
  "mcpServers": {
    "klit-flow": {
      "command": "klit-flow",
      "args": ["serve"]
    }
  }
}
```

### 4. Optional: generate NL summaries

Requires a running Ollama instance with a model pulled (e.g. `ollama pull llama3.2`):

```bash
klit-flow analyze /path/to/my-android-app --platform android --summaries
```

Each module and screen doc gains a 1–2 sentence description in its body.

## Troubleshooting

| Problem | Fix |
|---------|-----|
| `klit-flow: command not found` | Add pip's bin dir to PATH: `export PATH="$HOME/Library/Python/3.11/bin:$PATH"` |
| `ModuleNotFoundError: No module named 'torch'` | `pip3 install torch` |
| Slow first `analyze` | Normal — embedding model is downloading; subsequent runs are instant |
| `FileNotFoundError: No index found` on `serve` or `query` | Run `klit-flow analyze` first to build the index |

## Output artifacts

All artifacts are written to `.klit-flow/out/` inside the target repo:

| Path | Description |
|------|-------------|
| `graph.json` | Full graph — `{"nodes": [...], "edges": [...]}` |
| `docs/modules/<name>.md` | One file per source module with YAML frontmatter |
| `docs/screens/<name>.md` | One file per screen with navigation frontmatter |
| `diagrams/dependencies.mmd` | Mermaid flowchart of module imports |
| `diagrams/flows.mmd` | Mermaid flowchart of screen navigation |

Frontmatter makes these dual-purpose: humans read the body, RAG pipelines index the structured fields.

## MCP tools reference

| Tool | Input | What it returns |
|------|-------|-----------------|
| `query(text)` | Natural-language query | Ranked list of matching nodes |
| `context(symbol)` | Symbol name | Inbound + outbound edges |
| `impact(symbol)` | Symbol name | Direct importers and callers |
| `flows(screen?)` | Screen name (optional) | NAVIGATES\_TO edges |
| `cypher(q)` | Cypher string | Raw query result rows |

## Architecture

The indexing pipeline runs in order:

```
walk → parse → resolve → flows → persist → index → emit
```

| Phase | Module | Description |
|-------|--------|-------------|
| Walk | `walker.py` | Discovers source files; honours `.gitignore` + `.klit-flowignore` |
| Parse | `parsing/` | tree-sitter per file; `.scm` query files per language; emits `Symbol` records |
| Resolve | `graph/resolver.py` | Turns imports + call sites into typed graph edges |
| Flows | `flows/` | Platform-specific ABC; adds `Screen` nodes and `NAVIGATES_TO` edges |
| Persist | `graph/store.py` | `GraphStore` ABC backed by LadybugDB (`import ladybug`) |
| Index | `index/` | Local `sentence-transformers` embeddings + BM25; hybrid RRF fusion |
| Emit | `emit/` | `graph.json`, Markdown docs, Mermaid diagrams |

The `GraphStore` interface is the only persistence boundary — swapping the storage engine requires changes only inside `store.py`.

---

## Constraints

- The `analyze` pipeline is **deterministic**: zero LLM calls, zero network access.
- Embeddings run **locally** via `sentence-transformers` (default model: `BAAI/bge-small-en-v1.5`).
- NL summaries use a **local Ollama model only** and are **off by default**.
- `IMPORTS` edges are exact. `CALLS` and `NAVIGATES_TO` edges are best-effort and carry an honest `confidence` score.
- Parse failures are logged and skipped — never fatal.
